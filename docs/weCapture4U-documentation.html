Chapter 1
Overview

Purpose & Scope

weCapture4U is a full-stack photography business management system built for a solo photographer (owner). It covers appointment scheduling, job lifecycle tracking, client management, client-facing portal, and a complete double-entry accounting system — all in a single application.

The frontend is a single React 18 + TypeScript SPA serving two role-based modules at separate route trees: the Admin module (photographer dashboard) and the Client Portal (client self-service). Both share the same FastAPI backend, PostgreSQL database, and JWT auth system.

Out of scope for this iteration: image upload pipeline (AWS S3), subscriptions/billing, multi-user admin, recurring appointments, mobile app, online payment collection, calendar sync, data import, contract/document generation.

Auth Strategy — Password + JWT Refresh

FastAPI owns all authentication — no Supabase Auth, no RLS. Credentials are stored in the users table with bcrypt-hashed passwords.

  • Login: POST /api/auth/login → FastAPI verifies credentials → returns access_token (JWT, 15 min expiry) + refresh_token (opaque, 7 days). Both stored in localStorage.
  • Token transport: Authorization: Bearer <access_token> via shared Axios interceptor on every request.
  • Silent refresh: On 401 response, the Axios interceptor automatically calls POST /api/auth/refresh → receives new access token → retries original request. If refresh fails, clears both tokens and redirects to login.
  • Route guards: <AdminRoute> and <ClientRoute> decode the token, check role, and redirect to login if missing or wrong role.
  • Deactivated accounts: users.is_active = false → 403 at login — "Your account has been deactivated."

Auth Strategy — Biometric (WebAuthn / FIDO2)

Biometric login is an optional second method layered on top of the existing JWT flow. The W3C WebAuthn standard is used — the device biometric (Face ID, Touch ID, fingerprint) never leaves the device. The server stores only a public key.

Registration

  1. User logs in with password for the first time.
  2. Prompted on the biometric setup page: "Enable Face ID / fingerprint for faster login?"
  3. Browser calls navigator.credentials.create() → device generates key pair in secure enclave → public key + credential ID sent to server → stored in webauthn_credentials.

Login

  1. User enters email on login page.
  2. GET /api/auth/webauthn/device-check?email=... → returns has_credential: bool.
  3. If true, "Use Face ID / Fingerprint" button appears alongside the password form.
  4. Server sends random challenge → navigator.credentials.get() → device signs challenge via biometric → signature sent to server → FastAPI verifies with stored public key → JWT issued.

The webauthn_credentials table and all WebAuthn endpoints are shared between admin and client users. Role is determined from the JWT issued after successful verification.

Password Reset Flow

Shared endpoints for both admin and client roles. The role is determined by the token's associated user.

  1. "Forgot password?" link on login page → enter email → POST /api/auth/forgot-password
  2. FastAPI generates a secure random token, stores SHA-256 hash in password_reset_tokens (expires 1 hour), sends reset email via Resend with link to /reset-password?token=<raw_token>
  3. User submits new password → POST /api/auth/reset-password → FastAPI validates token (not expired, not used), hashes new password, marks token used
  4. Redirected to login. Expired/used token → 422 — "This reset link is invalid or has expired."

Email & Scheduled Jobs

APScheduler runs within the FastAPI process. A daily job fires at 08:00 UTC and:

  • Queries appointments starting within 24h → inserts notification rows + sends reminder email
  • Queries clients with birthday matching today → birthday alert
  • Queries invoices past due date → overdue alert

Missed runs (e.g., server downtime) are not recovered — accepted behavior for this iteration. Emails sent via Resend. Email send failures do not crash the app — logged server-side, graceful degradation.

Automation Rules

Every business event generates a draft journal entry. The admin must explicitly post each entry. Nothing posts silently.

TriggerAuto-generated Draft Entry
Invoice status → sent Dr Accounts Receivable (total) · Cr [revenue account per invoice_items, defaults to Session Fees] (subtotal − discount) · Cr Tax Payable (tax). Tax Payable line omitted when tax = 0.
Payment recorded (invoice_payments created) Dr [payment.account_id] (amount) · Cr Accounts Receivable (amount). Fires for every payment: partial, full, or deposit catch-up.
Deposit received (appointments.deposit_paid → true) Dr [deposit_account_id, defaults to 1010] (deposit_amount) · Cr Deferred Revenue (deposit_amount). Only fires if deposit_amount > 0.
Invoice created from appointment with posted deposit Dr Deferred Revenue (deposit_amount) · Cr Accounts Receivable (deposit_amount). Also creates invoice_payments record (amount = deposit_amount, notes = "Deposit applied") to keep balance_due in sync.
Expense created (payment_status = paid) Dr [expense_account_id] (amount) · Cr [payment_account_id] (amount)
Expense created (payment_status = payable) Dr [expense_account_id] (amount) · Cr Accounts Payable (amount)
Expense paid (payable → paid) Dr Accounts Payable (amount) · Cr [admin selects: Business Bank Account / Cash on Hand] (amount)

Manual entry only (no automation): Loan repayments · Owner drawings · Owner capital injections · Investment account movements · Depreciation · End-of-year retained earnings transfer · Corrections and adjustments

Journal Entry Review Flow

  1. Admin triggers a business event → a slide-over panel opens showing the auto-generated draft
  2. Panel shows: date (editable, admin can backdate), description (editable), reference link, debit/credit lines table, balance indicator (Debits: €X · Credits: €Y · Difference: €0.00 — red if non-zero)
  3. Admin can edit any line, add new lines, remove non-mandatory lines
  4. "Post Entry" — validates balance, saves as posted, affects all reports immediately
  5. "Save as Draft" — saves without posting; remains visible in Journal tab and Overview banner
  6. "Discard" — cancels; originating event (expense, invoice status) is still committed. Admin can create entry manually later.
  7. Voiding a posted entry: confirmation → reversing entry created (all amounts swapped, dated today, void_of = original_id) → new corrected draft opens. If voided entry was a payment entry, linked invoice gets requires_review flag.

Key metric cards: Total Revenue · Total Expenses · Net Profit · Cash Position (Bank + Cash on Hand) · Outstanding AR · Loan Balance. Revenue vs Expenses bar chart (Recharts, filterable by year). Recent posted entries (last 5). Draft entries alert banner when drafts exist.

Paginated list of all journal entries (posted + draft + voided), ordered by date desc. Columns: date, description, reference badge, status badge, total debit, created by. Click any row to expand and see all debit/credit lines. Filters: status, date range, reference type, account. "New Journal Entry" button → blank manual entry slide-over. Void button on posted entries.

Tab 3 — Accounts

Chart of accounts grouped by type (Assets → Liabilities → Equity → Revenue → Expenses). Each row: code, name, current balance (from posted lines), normal balance indicator. Add account (code, name, type). Rename inline. Archive for non-system accounts. Click any account → Account Ledger: all posted lines ordered by date, opening/running/closing balances.

Tab 4 — Invoices

Existing invoice list. Marking an invoice as paid opens payment recording form (amount, date, account) → triggers auto-draft journal entry. Invoices with requires_review flag show warning badge with tooltip.

Chapter 7
Client Portal

The client portal is the private dashboard used by photography clients to track job progress, access delivered photos, and submit booking requests. Mobile-first; shares the FastAPI backend, PostgreSQL database, and JWT auth with the admin module.

Role enforcement: All client endpoints check role == 'client'. Clients can only read their own data — all queries filtered by client_id derived from the authenticated user. Attempts to access another client's resource return 403.

Visual Style

Dark base (#0c0c0c) with amber-tinted cards (rgba(245,158,11,0.08) background, rgba(245,158,11,0.2) border). Mobile-first responsive — base styles target < 768px; tablet/desktop via min-width breakpoints.

Layout: Top bar (logo left, client avatar right) + bottom navigation on mobile (Home · My Jobs · Bookings · Profile). Desktop: top nav with same links inline.

Auth

  • Client accounts created by admin (not self-registration). Admin creates users and clients rows atomically; client receives email with login credentials.
  • Same JWT refresh flow as admin — access_token (15 min) + refresh_token (7 days). Axios interceptor handles silent refresh on 401. If refresh fails → redirected to /client/login.
  • Same password reset flow (shared endpoints). Link goes to /client/reset-password?token=....
  • Same WebAuthn biometric flow — registration prompt after first password login via /client/biometric/setup. GET /api/auth/webauthn/device-check determines whether to show "Use Face ID / Fingerprint" button.
  • Deactivated account (users.is_active = false) → 403 — "Your account has been deactivated. Contact your photographer."

Dashboard (/client)

  • Active job card: Most recent job where stage is not terminal. Shows job title, current stage + amber dot, progress bar. "You have X active jobs" link if more than one.
  • Deposit status card: appointments.deposit_paid from linked appointment. Hidden if no linked appointment.
  • Next appointment: Nearest future appointment — date, time slot, location.
  • Book a New Session CTA: Amber button → /client/bookings/new

CORS Configuration

FastAPI CORS middleware. Origins loaded from ALLOWED_ORIGINS env var (split on comma). In development, also allow http://localhost:5173.

  • allow_origins: list parsed from env var
  • allow_credentials: True — required for Authorization header
  • allow_methods: ["*"]
  • allow_headers: ["*"]

Never hardcode origins. If ALLOWED_ORIGINS is missing in production, FastAPI must raise a startup error — not silently allow all origins. A misconfigured CORS policy is a security vulnerability.

Health Check Endpoint

FastAPI must expose GET /api/health → returns {"status": "ok"} with HTTP 200. No auth required. Used by Fly.io health checks to determine if the machine is ready to serve traffic.

Frontend Build & Deploy (Hostinger)

  1. Set VITE_API_URL in .env.production to the Fly.io backend URL.
  2. Run npm run build → produces dist/ folder.
  3. Upload contents of dist/ to Hostinger via FTP or File Manager into public_html/.
  4. Add .htaccess in public_html/ for React Router SPA support:
Options -MultiViews
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteRule ^ index.html [QR,L]

LAUNCH BLOCKER — .htaccess is required. Without it, Hostinger's Apache server returns a 404 for any URL that isn't the root. Refreshing /client/jobs/123 or navigating directly to /admin/accounting will show a blank error page — the app will appear completely broken to users. This file must be present in public_html/ alongside index.html before going live.

HTTPS: Hostinger provides free SSL via Let's Encrypt — enable in Hostinger panel under SSL/TLS.

WebAuthn requires HTTPS. Biometric login (WebAuthn / FIDO2) will not function on plain HTTP. Both the frontend (Hostinger) and backend (Fly.io) must be served over HTTPS before testing biometric registration or authentication in production.

Backend Deploy (Fly.io)

Initial Setup

  1. Install Fly CLI: brew install flyctl (macOS)
  2. Login: flyctl auth login
  3. From project root: flyctl launch → select region, name the app (e.g. wecapture4u-api). Do not create a Fly Postgres — using Supabase.
  4. Set secrets: flyctl secrets set DATABASE_URL="..." JWT_SECRET_KEY="..." RESEND_API_KEY="..." ... (all env vars above)
  5. Deploy: flyctl deploy
  6. Check logs: flyctl logs

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .

fly.toml Template

app = "wecapture4u-api"
primary_region = "fra"  # Frankfurt — close to Europe

[build]

[http_service]
  internal_port = 8000
  force_https = true
  auto_stop_machines = false
  auto_start_machines = true
  min_machines_running = 1

[[vm]]
  memory = "256mb"
  cpu_kind = "shared"
  cpus = 1

LAUNCH BLOCKER — auto_stop_machines = false is required. Without this, Fly.io puts the machine to sleep after inactivity. APScheduler runs inside the FastAPI process — when the machine sleeps, the scheduler stops entirely. Appointment reminders, birthday alerts, and invoice overdue notifications will silently never fire. Set auto_stop_machines = false and min_machines_running = 1 in fly.toml before the first deploy and never change them.

  2. Run against Supabase using the Supabase SQL editor or psql with the connection string.
  3. Always run migrations before deploying new backend code that depends on schema changes.
